# Архитектура экосистемы

```text
Сайт ───────────────┐
                    ├── ITMO Megabattle Backend ── Supabase/Postgres + Storage
Бот участников ─────┤              │
  └─ Telegram Mini App             ├── ITMO.ID
                                   └── ITMO Events

Бот организаторов — автономный проект вне этой схемы.
```

Backend обслуживает ровно два клиента: сайт и бот участников. У бота организаторов нет service token, маршрутов, таблиц, фоновых задач или доступа к общей базе.

## Идентичность

`profiles.id` — внутренний неизменяемый ID человека. `account_identities` привязывает к нему Telegram ID, а позднее — ITMO.ID subject. Supabase Auth не используется. Один человек сможет входить обоими привязанными способами и попадёт в тот же профиль.

Telegram сейчас является основным входом. После выдачи credentials ITMO.ID подключается без миграции профилей. Для мероприятий с `requires_itmo_id=true` backend возвращает `428 ITMO_ID_REQUIRED`, пока пользователь не привяжет ITMO.ID.

## Роли

- `participant` — участник;
- `admin` — администратор экосистемы;
- `site_admin` — административный доступ к сайту.

Внутренние роли и процессы организаторов не являются частью этой модели.

## Фоновые процессы

Worker:

- создаёт и отправляет участникам напоминания о мероприятиях;
- синхронизирует регистрации с ITMO Events;
- удаляет временные фото;
- обрезает аудит до 50 записей.

На VPS используется `WORKER_ENABLED=true`. На serverless вызывается `POST /internal/cron` внешним cron с `CRON_SECRET` каждые 1–5 минут.

## Безопасность

- Service Role, токен бота участников, ITMO credentials и vault key находятся только в environment backend;
- Telegram Mini App `initData` проверяется HMAC;
- OIDC использует signed state, nonce, discovery и проверку ID Token через JWKS;
- callback ITMO.ID возвращает одноразовый пятиминутный code, а не токен в URL;
- роль и бан перепроверяются в БД на каждом защищённом запросе;
- балансы изменяются только транзакциями; переводы, reward code и командная регистрация выполняются атомарными SQL-функциями;
- очередь захватывает записи через `FOR UPDATE SKIP LOCKED`, поэтому параллельные worker-функции не дублируют отправку;
- пароли шифруются AES-256-GCM, PIN используется как второй барьер доступа;
- временные фото лежат в private bucket и отдаются по короткой signed URL.
